# Toloka — ручная адъюдикация конфликтов разметки reasoning-событий

Статический вьюер (GitHub Pages) для того, чтобы глазами отсмотреть места, где разные
автоматические разметчики («агенты») reasoning-событий расходятся, вынести человеческий
вердикт и по итогам получить score каждого агента для ранжирования примеров.

Агенты: `regex` (детектор-регулярки), `Claude`, `DeepSeek`, `Qwen`.

## Как это работает в двух словах

1. Открываешь страницу Pages, вписываешь свой ник в поле «аннотатор».
2. Листаешь конфликтные сайты. На каждом видно, что сказал каждый агент (тип события или
   ∅ «не сработал») и его цитата-триггер в контексте трассы. Контекст подгружается по мере
   скролла вверх/вниз (окно ±12 сегментов сразу, дальше — по прокрутке до всей трассы).
3. Ставишь вердикт: какой тип события здесь на самом деле. Можно выбрать НЕСКОЛЬКО типов
   (мультиселект — клик по типу добавляет/снимает его), либо ∅ (события нет), либо «неясно»
   (∅ и «неясно» — исключающие). Клавиши: `1-9` типы, `0` = ∅, `u` = неясно, `j`/`k` навигация.
4. Сохраняешь результат одним из двух способов (см. ниже).
5. `python3 build/score_agents.py` считает score агентов и согласие разметчиков.

Разметка живёт в localStorage браузера и между сессиями не теряется.

## Как сохранить результат

Два пути:

1. Файлом: «экспорт разметки» -> скачивается `annot_<ник>.json` -> кладёшь в `annotations/` ->
   `git commit && git push`.
2. Коммитом из браузера (кнопка «💾 GitHub»): вписываешь owner/repo/branch/path и
   fine-grained PAT с правом `contents:write`, жмёшь «коммитить разметку» — файл
   `<path>/annot_<ник>.json` уходит прямо в репозиторий через api.github.com. Работает и с
   ПРИВАТНЫМ репо, если у токена есть к нему доступ. Токен хранится только в твоём браузере
   (localStorage) и никуда, кроме api.github.com, не отправляется; в саму страницу он не зашит.
   По умолчанию цель — `karpovilia/toloka`, поменяй на свой приватный репо при желании.

## Реальность данных (важно)

Настоящего 4-угольника на ОДНИХ И ТЕХ ЖЕ трассах нет: разметки делались на разных сэмплах.

- Claude и DeepSeek пересекаются на 1693 трассах (модели gemma / qwen) — здесь честная тройка
  агентов `regex + Claude + DeepSeek` (срез A).
- Qwen размечал только gpt-oss-трассы (485 штук) и НЕ пересекается ни с Claude, ни с DeepSeek —
  на его трассах доступна только пара `regex + Qwen` (срез B).
- Пересечение всех четырёх на одной трассе = 0.

Поэтому вьюер N-угольный: на каждой трассе показаны ровно те агенты, кто её реально разметил.
Каждый агент скорится на своём подмножестве конфликтов (Qwen — на срезе B, остальные — на A).

## Что такое «конфликтный сайт»

События агентов выравниваются по сегментам (окно ±1). Сайт — кластер событий рядом; на нём у
каждого присутствующего агента есть ответ (тип или ∅). Сайт попадает в корпус, только если
ответы агентов не совпадают. Полное согласие в разметку не выносится.

Приоритет (для порядка разметки):

- P4 — разные типы у двух+ LLM-агентов (самое ценное);
- P3 — один LLM поставил событие, другой молчит;
- P2 — LLM расходится с regex;
- P1 — одиночное срабатывание (кандидат в FP regex).

Во вьюер уходит стратифицированный топ (по умолчанию 6000 сайтов): квоты по срезам A/B и по
тирам приоритета, чтобы адъюдицировались все режимы ошибок и все агенты получили score.
Полный список всех ~196k сайтов — в `build/sites_full.jsonl` (не публикуется, регенерируется).

## Скоринг агентов

`build/score_agents.py` по файлам `annotations/annot_*.json`:

- gold на сайте = мажоритарный вердикт разметчиков; «неясно» исключается;
- accuracy агента = доля сайтов, где его ответ содержит gold-тип (± Wilson 95% CI);
- per-type precision / recall;
- межаннотаторное согласие: Cohen κ (ровно 2 разметчика) или Fleiss κ (≥3);
- Dawid–Skene (EM) — надёжности агентов и мягкие метки по всем сайтам;
- `data/next_batch.jsonl` — следующая пачка на разметку (топ неразмеченного по приоритету и
  разногласию).

Важная оговорка: метрики условны НА конфликтных сайтах — выборка не случайна, это относительное
ранжирование агентов, а не абсолютный recall/precision на всём корпусе.

Про κ: Cohen κ по определению для двух оценщиков; для трёх и более — Fleiss κ. Скрипт выбирает
сам по числу разметчиков.

## Рабочий процесс «сохранить результат в GitHub»

    # разметил -> «экспорт разметки» -> скачался annot_<ник>.json
    cp ~/Downloads/annot_<ник>.json annotations/
    git add annotations/annot_<ник>.json
    git commit -m "разметка <ник>: +N сайтов"
    git push
    python3 build/score_agents.py     # обновить score_report.md

## Структура

    index.html, styles.css, app.js   вьюер (чистая статика)
    config.json                      пути к данным
    data/conflicts.json              сайты для вьюера (self-contained, с контекстом ±12)
    data/event_types.json            модель типов событий (домены M/R)
    data/build_summary.json          статистика сборки
    data/score_report.md             отчёт по score (обновляется скриптом)
    annotations/                     сюда кладут annot_<ник>.json
    build/build_conflicts.py         сборка корпуса из исходных разметок
    build/score_agents.py            подсчёт score
    build/test_viewer_smoke.js       jsdom-смоук вьюера

## Пересборка корпуса

    python3 build/build_conflicts.py --cap 6000 --radius 12

Читает исходные разметки из `internal_signals_poc/{gold,gold_deepseek_dual,gold_qwen35_dual}`
и `payloads/`, regex пересчитывает тем же детектором, что и `build_verification.py`.
